# Adaptive Firmware Layer — Sharp Thesis

_Generated by benchmarks/analysis/run_all.py — four runtime-validated analyses_

## The core claim, falsifiably stated

> **Adaptive hardware reconfiguration provides a statistically significant advantage**
> **over static compilation IF AND ONLY IF the reward spread between accelerator**
> **configs for different workload phases exceeds the reconfiguration overhead.**
>
> When configs are clearly differentiated (compute vs memory-optimized), adaptation
> dominates at _any_ reconfiguration cost. When configs overlap in capability,
> static wins — because the cost of exploration exceeds the marginal improvement.

This is a different claim from the literature, which focuses on workload heterogeneity
as the key driver. We find that heterogeneity alone is insufficient. Config separation
matters more.

## Evidence

### 1. Synthetic workload: adaptive dominates unconditionally

We generated workloads with controlled switch probability (0%–100%) between two
clearly separated classes: compute_bound (AI ≈ 30, optimal: HIGH_COMPUTE) and
memory_bound (AI ≈ 2, optimal: HIGH_BANDWIDTH).

| Switch Prob | Best Adaptive | Best Static | Delta | Winner |
|---|---|---|---:|---:|---|
| 0% | 0.9777 (smart_static) | 0.7684 (static_2) | +0.2093 | adaptive |
| 1% | 0.7738 (tabular) | 0.6063 (static_2) | +0.1675 | adaptive |
| 5% | 0.7077 (tabular) | 0.5494 (static_2) | +0.1583 | adaptive |
| 10% | 0.7510 (tabular) | 0.5795 (static_2) | +0.1715 | adaptive |
| 30% | 0.6952 (tabular) | 0.5842 (static_2) | +0.1110 | adaptive |
| 100% | 0.7204 (tabular) | 0.5673 (static_2) | +0.1531 | adaptive |

**Key finding**: Adaptive beats static at _every_ heterogeneity level, including 0%.
Even at switch_prob=0% (all traces identical), smart_static (0.9777) crushes
static_2 (0.7684) because it profiles once and picks the _right_ fixed config.

**Why?** The reward gap between HIGH_COMPUTE and BALANCED on a compute-bound workload
is large (~0.98 vs ~0.77). Even a single profiling pass pays for itself. The
reconfiguration overhead (~8ms) is dwarfed by the per-trace execution time improvement.

**Implication**: When configs are well-differentiated, adaptation always wins.
The break-even threshold is at config separation, not at workload heterogeneity.

### 2. Real workload (mixed_production): static LOW_POWER wins — narrowly but significantly

We ran 15 seeds each of tabular, smart_static, static_3, and oracle on the
mixed_production workload (40% LLM decode, 25% CV, 20% audio, 15% LLM prefill).

| Agent | Mean Reward | Std | 95% CI | vs static_3 |
|---|---|---|---|---|
| static_3 | **0.3866** | 0.0036 | ±0.0018 | — |
| oracle | 0.3738 | 0.0089 | ±0.0045 | **p = 0.0000** (worse) |
| tabular | 0.3715 | 0.0203 | ±0.0103 | **p = 0.0047** (worse) |
| smart_static | 0.3419 | 0.0562 | ±0.0284 | p = 0.0551 (n.s.) |

**Key finding**: static_3 (LOW_POWER) beats every adaptive agent on mixed_production.
The difference is statistically significant (p < 0.01).

**Why?** On mixed_production, all four accelerator configs achieve similar throughput
(within 2x) on most phases. The LOW_POWER config (4 pJ/op) is "good enough" at
everything and never pays reconfiguration overhead. The adaptive agents waste ~40%
of traces on exploration and reconfiguration.

**Implication**: Even on a genuinely heterogeneous workload with 4 model types,
the best static config wins when the configs aren't sharply differentiated for
the specific workload phases.

### 3. llm_decode: adaptation is strictly harmful

| Agent | Mean Reward | Std | 95% CI | vs static_3 |
|---|---|---|---|---|
| static_3 | **0.3690** | 0.0000 | ±0.0000 | — |
| smart_static | 0.3457 | 0.0000 | ±0.0000 | **p = 0.0000** |
| oracle | 0.3354 | 0.0000 | ±0.0000 | **p = 0.0000** |
| tabular | 0.2893 | 0.0256 | ±0.0129 | **p = 0.0000** |

**Key finding**: Even the oracle (with perfect knowledge) loses to static_3.
Zero standard deviation for deterministic agents — these are not measurement errors.

**Why?** llm_decode is homogeneous: all traces are memory-bound, with growing
KV cache. The optimal config (LOW_POWER) never changes. Any reconfiguration
is pure waste.

**Implication**: On homogeneous workloads, adaptation is _strictly harmful_.
A compile-time compiler (TVM/TensorRT) would correctly profile once and pick
the best static config. Adaptive agents would need a "detect homogeneity and
stay put" trigger — which they currently lack.

### 4. Reconfiguration cost: not the bottleneck we expected

On synthetic heterogeneous workload (switch_prob=30%), we swept reconfig time
from 0.05x to 50x of baseline values:

| Mult | Best Adaptive | Best Static | Delta | Winner |
|---|---:|---:|---:|---:|
| 0.05x | 0.7510 | 0.5843 | +0.1666 | adaptive |
| 1.0x | 0.7456 | 0.5842 | +0.1614 | adaptive |
| **50x** | **0.7363** | **0.5842** | **+0.1522** | adaptive |

**Key finding**: Even at 50x reconfiguration time (400ms for HIGH_COMPUTE),
adaptive still beats static on a heterogeneous workload.

**Why?** When configs are sharply differentiated, the per-trace reward improvement
from picking the correct config (e.g., HIGH_COMPUTE over BALANCED on a compute-bound
trace) exceeds even 400ms of reconfiguration overhead amortized over ~500 traces.

**Implication**: Reconfiguration cost is a second-order concern. The primary
determinant of whether adaptation pays off is **config differentiation** —
how much reward is left on the table by picking the wrong config.

### 5. Oracle gap: the current oracle is naive

The greedy oracle (switch on every workload class change) underperforms static_3
on mixed_production and llm_decode because it ignores reconfiguration cost:

| Workload | Best Agent | Oracle Reward | Best Static | Gap (meaningful) |
|---|---|---|---|---|
| audio_encoder | static_2 | 0.6795 | 0.7076 | N/A (oracle < static) |
| cv_detection | static_2 | 0.6458 | 0.6458 | N/A (tie) |
| llm_decode | static_3 | 0.3354 | 0.3690 | N/A (oracle < static) |
| llm_prefill | oracle | 0.6720 | 0.5626 | **oracle wins (+19.4%)** |
| mixed_production | static_3 | 0.3668 | 0.3835 | N/A (oracle < static) |

**Key finding**: The oracle only beats static on llm_prefill — the one workload
where compute-bound phases are distinct enough that switching pays off. On all
other workloads, the greedy oracle's reconfiguration cost exceeds its benefit.

**Implication**: A truly optimal dynamic policy would trade off switching vs
staying based on expected phase duration. This is a look-ahead optimization,
not a per-step greedy choice. The current oracle is a _lower bound_ on what a
smart dynamic policy could achieve.

## Synthesis: the real thesis

**The viability of adaptive reconfiguration is determined by the reward spread
between accelerator configs, not by workload heterogeneity.** When config
separation is large (synthetic workloads), adaptation dominates at any
reconfiguration cost and any heterogeneity level. When config separation is
small (real workloads), static wins — the exploration and switching overhead
exceeds the marginal improvement.

The practical implication: **adaptive reconfiguration is most valuable for
heterogeneous compute workloads running on hardware where different phases
map to sharply different optimal configs.** This is likely true for:
- Mixed-precision neural networks (INT8 vs FP16 vs FP32 phases)
- Multi-model serving (LLM decode + vision + audio on the same accelerator)
- Sporadic compute bursts (serverless + batch inference on shared hardware)

For a typical AI serving workload with reasonably energy-efficient hardware,
a static LOW_POWER config is good enough — and the adaptive firmware layer
needs sharper config differentiation to justify its overhead.

## Open questions (for real hardware)

1. **Config discrimination metric**: Can we predict the "config separation" of
   a hardware platform (reward spread between configs across workload classes)
   from its roofline parameters alone?

2. **Phase-aware oracle**: What's the optimal switching policy when reconfiguration
   has a cost and phase durations are unknown? This is a classic explore-exploit
   problem with switching cost — the Gittins index solution.

3. **Config design**: If we could design accelerator configs from scratch,
   would we maximize separation (each config specialized for one class) or
   minimize it (each config good at everything)? The thesis suggests
   separation — but real hardware constraints may push the other way.

4. **Simulation validity**: The roofline model assumes perfect parallelization.
   Real hardware has pipeline bubbles, memory hierarchy effects, and thermal
   constraints that may change the config separation landscape.
